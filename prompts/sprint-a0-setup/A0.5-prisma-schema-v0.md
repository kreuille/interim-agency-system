## Contexte
Pose le modèle relationnel qui porte l'ensemble du SI. Toutes les tables sont définies ici pour éviter les migrations "grand chantier" plus tard (on ajoute des colonnes, pas des tables).

## Principes de conception
- UUID v4 partout.
- Timestamptz UTC.
- Montants en `BigInt` (Rappen).
- Énumérations Prisma pour les statuts figés.
- FK `ON DELETE RESTRICT` pour les données légales.
- Index composites avec `agency_id` en tête.

## Livrable clé — schema.prisma (extrait)

```prisma
generator client { provider = "prisma-client-js" }
datasource db { provider = "postgresql"; url = env("DATABASE_URL") }

model Agency {
  id          String   @id @default(uuid()) @db.Uuid
  legalName   String
  ideNumber   String   @unique
  canton      String   @db.Char(2)
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt
  workers     TempWorker[]
  clients     Client[]
  authorizations LseAuthorization[]
  @@map("agencies")
}

model TempWorker {
  id              String   @id @default(uuid()) @db.Uuid
  agencyId        String   @db.Uuid
  firstName       String
  lastName        String
  avs             String
  iban            String
  residenceCanton String   @db.Char(2)
  email           String?
  phone           String?
  employmentType  EmploymentType @default(TEMP_WORKER)
  createdAt       DateTime @default(now())
  updatedAt       DateTime @updatedAt
  archivedAt      DateTime?
  agency          Agency   @relation(fields: [agencyId], references: [id])
  @@unique([agencyId, avs])
  @@index([agencyId, archivedAt])
  @@map("temp_workers")
}

// ... autres modèles détaillés à générer
```

## Tests
- Migration SQL valide (pas d'erreur PostgreSQL).
- Seed insère et se relit.
- Middleware tenant : insert sans agency_id → throw; select cross-agency → 0 résultat.

## Dette acceptable pour v0
- Colonnes qui pourraient être ajoutées plus tard : `reliabilityScore`, `consent_snapshot`, `gdpr_anonymized_at`.
- Pas encore d'optimisation partitioning (timesheets). À faire quand > 1M lignes.
