## Contexte

C'est le moment critique où une agence cliente potentielle soit s'engage, soit abandonne. Il faut que ce flow :
- Soit **rassurant** : le user voit ce qui a été validé, ce qu'il reste.
- Soit **tolérant aux interruptions** : pause + reprise demain sans perte.
- Soit **conforme d'emblée** : sans LSE valide, impossible d'activer (règle B2.4).

## Étapes détaillées

### Étape 1 — Entité légale
- IDE (validation mod 11 réutilisée depuis `packages/shared/src/ide.ts`).
- Raison sociale exacte (vérifiée via API ZEFIX ou RCEnt si dispo, sinon saisie libre).
- Forme juridique (SA, Sàrl, SNC, raison individuelle).
- Canton du siège.
- Adresse complète.
- Date de création.

Si IDE valide + match ZEFIX : pré-remplir automatiquement la raison sociale et adresse (gain de temps).

### Étape 2 — Autorisation LSE
- Type : cantonale / fédérale.
- Numéro d'autorisation.
- Canton émetteur.
- Date d'émission + date d'expiration.
- **Upload PDF scan** de l'autorisation (CMEK chiffré dans bucket tenant).
- Validation manuelle par staff éditeur (ou OCR auto best-effort + review staff < 24h).

Règle bloquante B2.4 : sans cette étape validée, tenant reste `pending_activation`.

### Étape 3 — Coordonnées bancaires
- IBAN (validation mod 97 réutilisée).
- Nom banque (auto-détecté depuis IBAN si possible).
- Devise compte : CHF uniquement pour MVP.
- Usage : compte qui reçoit les paiements clients + compte utilisé pour virements intérimaires pain.001.

### Étape 4 — Branding (skippable)
- Upload logo (SVG ou PNG 512+).
- Couleur primaire (picker).
- Couleur accent (picker, optionnel).
- Tagline ou slogan (200 chars max).

Si skip : branding par défaut Helvètia Intérim + co-branding "powered by Helvètia Intérim".

### Étape 5 — DPA signature ZertES
- Affichage DPA template v1 (B0.5) personnalisé avec les infos tenant de l'étape 1.
- Signature électronique avancée via Swisscom Trust Services.
- PDF signé archivé dans GED tenant, hash chain audit log.
- Bloquant : pas d'étape 6 sans DPA signé.

### Étape 6 — Paiement Stripe
- Choix du plan (Starter / Pro / Enterprise).
- Choix facturation mensuelle ou annuelle (-15%).
- Stripe Checkout embedded ou redirect.
- Capture du Customer + Subscription Stripe, sauvegarde `stripe_customer_id` et `stripe_subscription_id` sur `Tenant`.
- Trial 14 jours par défaut (paramétrable par code promo).
- Webhook `checkout.session.completed` → activation tenant `status: active`.

### Étape 7 — Invite team (skippable)
- Formulaire multi-lignes : email + rôle (dispatcher, payroll_officer, hr, sales, viewer).
- Emails d'invitation envoyés automatiquement (magic link avec création compte dans le tenant).
- Skippable, peut être fait plus tard depuis `/settings/users` (B1.5).

## Tests

- E2E Playwright : parcours nominal complet (7 étapes).
- Interruption à étape 3 → reconnexion → reprise à étape 3 avec données étape 1-2 préservées.
- Skip étape 4 et 7 → tenant activable.
- Skip étape 2 (LSE) → tenant pas activable.
- Upload LSE fichier > 10 MB → erreur claire.
- IDE invalide mod 11 → message d'erreur.
- Paiement Stripe refusé → error + possibilité de retry.
- A11y : Axe sur chaque étape zero critical/serious.

## Risques

| Risque | Mitigation |
|--------|-----------|
| User abandonne au milieu | Email relance J+1, J+3, J+7 ; CSM suivi manuel pour gros prospects |
| LSE uploadée fausse/expirée | Validation staff manuelle obligatoire avant activation |
| Paiement Stripe échoue 3x | Tenant reste en trial mais bloqué, email support |
