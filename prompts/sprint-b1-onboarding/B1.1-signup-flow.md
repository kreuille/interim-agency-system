## Contexte

Ce flow est la **première impression produit**. Il doit être friction minimum tout en bloquant les abus (bots, tenants fantômes). La création de **tenant Firebase** est la partie délicate : elle nécessite les credentials Admin SDK + la Cloud Function `onTenantCreate` (partie de DETTE-016 déjà identifiée).

## Instructions

1. **Page /signup** (Next.js RHF + Zod) avec champs :
   - email (validation RFC + domain MX check server-side)
   - legalName (1-100 chars)
   - firstName, lastName (1-100 chars)
   - phone (format +41XX... validé)
   - canton (select des 26)
   - acceptCgu (checkbox, obligatoire)
   - acceptDpa (checkbox, obligatoire)
   - captcha (Turnstile ou hCaptcha invisible)

2. **POST /api/signup** :
   - Valide le captcha.
   - Valide que l'email n'est pas déjà associé à un tenant actif (idempotency : retour 200 "check your email" même si exists, pour ne pas leaker).
   - Stocke en DB une `pending_signup` row avec token UUID + expiration 24h.
   - Envoie magic link email avec URL `https://app.helvetia-interim.guedou.ch/signup-callback?token=...`
   - Rate limit par IP + email.

3. **Callback magic link** :
   - Valide le token (signature + expiration + single-use).
   - Crée Firebase tenant via Admin SDK :
     ```typescript
     const tenant = await admin.auth().tenantManager().createTenant({
       displayName: legalName.slice(0, 20),
       emailSignInConfig: { enabled: true, passwordRequired: false },
       multiFactorConfig: { state: 'ENABLED', factorIds: ['phone', 'totp'] },
     })
     ```
   - Crée le compte user dans le tenant, role `agency_admin` via custom claims (trigger Cloud Function `onTenantCreate` = DETTE-016 à wiring fin).
   - Crée ligne `Tenant` en DB :
     ```typescript
     { id: uuid, firebaseTenantId, legalName, canton, status: 'onboarding', plan: 'trial', ownerEmail, createdAt }
     ```
   - Envoie event `TenantProvisioned` → audit log staff + welcome email.
   - Session user créée (Firebase ID token avec tenant scope), redirigée vers `/onboarding/mfa-setup`.

4. **MFA setup obligatoire** : page Next.js qui demande TOTP (authenticator app) ou SMS OTP. User bloqué sur les autres pages tant que MFA pas configurée.

5. **Anti-abuse** :
   - Captcha invisible côté client.
   - Rate limit Redis (par IP + email).
   - Détection patterns : emails jetables (MX check vs liste `disposable-email-domains`), noms trop courts, pattern bot.
   - Tenants en `status: onboarding` sans progression > 7 jours → suspendus auto (job hebdo).

## Tests

- Unit : provisionneur tenant, validation du form.
- Intégration : signup → magic link envoyé → callback → tenant créé en DB + Firebase.
- E2E Playwright : parcours complet avec Mailhog + Firebase emulator.
- Sécurité : tentative de signup 10x rapides → rate limit kick in. Token magic link réutilisé → refusé.
- Isolation : tenant créé, un autre signup → nouveau tenant ≠, cross-tenant impossible en DB.

## Risques et mitigations

| Risque | Mitigation |
|--------|-----------|
| Abus signup (spam tenants vides) | Rate limit + captcha + email verification + auto-suspend 7j |
| Email jetable | Liste `disposable-email-domains` + MX check |
| Fuite info "email déjà existant" | Retour générique "check your email" uniformément |
| Firebase tenant creation fail | Retry + rollback DB (pas de tenant orphelin ni compte orphelin) |
| Magic link phishing | Domaine signé + TLS strict + lien unique 15min exp |
