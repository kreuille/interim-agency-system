## Contexte

Tout Claude qui travaille après la bascule SaaS doit comprendre que :

1. Chaque modification peut affecter **plusieurs agences clientes distinctes** — pas juste "notre" agence.
2. L'audit des accès staff éditeur aux données tenants est **obligatoire contractuellement**.
3. Un bug de fuite cross-tenant n'est plus un inconvénient, c'est une **violation contractuelle + nLPD** avec obligation d'annonce PFPDT 72h.
4. Les données clients ne sont pas anonymisables librement (contrat de sous-traitance limite les usages).

Sans ces règles en tête, un Claude pourrait par exemple "optimiser une requête" en retirant le filter `agencyId` par mégarde — geste anodin en phase 1, catastrophique en phase 2.

## Amendements concrets à apporter

### Ajout §10 "Contexte éditeur SaaS (phase 2 post-pilote)"

```markdown
## 10. Contexte éditeur SaaS

À partir du pivot validé en ADR-0006, Helvètia Intérim est **éditeur SaaS** et opère **des agences clientes distinctes**. Règles supplémentaires :

### 10.1 Sous-traitance nLPD
- L'éditeur est **sous-traitant** au sens de la nLPD pour chaque agence cliente.
- Aucun usage des données tenant hors strict nécessaire au service (pas de profiling, pas d'entraînement IA sans consentement explicite).
- DPA signé avec chaque tenant — à ne jamais outrepasser.

### 10.2 Accès staff éditeur aux données tenant
- Tout accès lecture/écriture d'un staff éditeur à des données tenant **tracé** dans `audit_logs_staff` (séparé du `audit_logs` tenant-interne).
- Trace minimum : who, when, tenant_id, action, justification textuelle obligatoire.
- Accès lecture pour débug autorisé seulement sur ticket support ouvert ou après autorisation fondateur.
- Accès écriture sur données tenant interdit sauf correction d'incident documenté.

### 10.3 Isolation cross-tenant — niveau d'exigence
- Requête sans filter `agencyId` = bug critique, bloquant en CI (tenant-guard de A1.x doit être actif partout).
- Tests E2E de fuite cross-tenant (B2.2) rejoués à chaque PR touchant domaine ou repository.

### 10.4 Provisionnement tenant
- Création tenant uniquement via flow onboarding B1.1 (pas de création directe en DB sans trace).
- Suspension tenant sur `invoice.payment_failed` → data read-only pendant 7 jours puis archive.
- Suppression tenant : seulement après demande explicite client + 30 jours de délai + data export fourni (nLPD droit à l'effacement + obligation de conservation 10 ans pour données légales).

### 10.5 Communication avec les clients agences
- Aucune modification de flow utilisateur-facing sans préavis 14 jours aux clients.
- Versioning SemVer sur l'API publique tenant (`/api/v1/`, `/api/v2/`).
- Breaking change = nouvelle version majeure + période de cohabitation 90 jours minimum.
```

### Durcissement §3.5 existant

Ajouter à la section existante "Multi-tenant" :

```markdown
- **Tenant-guard Prisma** (via `$extends`) est **actif en prod** et doit lever une erreur runtime si une query arrive sans `agencyId` dans le context `AsyncLocalStorage`. Aucune exception.
- **Test de non-régression** : toute PR qui modifie un repository doit démontrer que le scénario "requête cross-tenant" échoue comme attendu.
- **Staff éditeur** : les utilisateurs avec le rôle `editor_staff` (nouveau, post B0.4) ont un accès technique multi-tenant, mais **tous leurs accès sont loggés dans `audit_logs_staff`**, jamais dans `audit_logs` tenant. Ces logs sont consultables par chaque tenant via un rapport "Qui a consulté mes données ?".
```

### Ajout §8 refus/escalade

```markdown
Claude **refuse** et escalade en contexte SaaS quand :

- Une demande implique de bypasser le tenant-guard "pour aller plus vite".
- Une demande implique de consulter des données tenant sans justificatif ticket.
- Une demande implique un export de données tenant hors du tenant concerné (sauf DPA ou ticket légitime).
- Une demande implique une modification silencieuse de CGU / pricing / DPA côté client sans avis préavis.
```
