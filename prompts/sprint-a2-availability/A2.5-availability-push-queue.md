## Schéma outbox
```
outbox_availability_push
  id uuid pk
  agency_id uuid not null
  worker_id uuid not null
  payload jsonb not null          -- slots à pousser
  idempotency_key uuid not null
  status enum(pending, in_progress, success, failed, dead)
  attempts int default 0
  next_attempt_at timestamptz null
  last_error text null
  created_at timestamptz
  @@index([status, next_attempt_at])
```

## Flow
1. Use case `UpdateAvailability` : mutation + insert outbox row (même tx).
2. Worker BullMQ : `SELECT ... FOR UPDATE SKIP LOCKED LIMIT 10`.
3. Appelle `AvailabilityPushPort.push()`.
4. Succès → status=success. Échec → attempts++, next_attempt_at=backoff.
5. Attempts > 6 → dead, DLQ alert.

## Tests
- Mutation + crash avant publish → au redémarrage, worker reprend et push.
- Push 2x même payload (rejeu) → idempotency store court-circuite le 2e appel.
- 500 × 3 → retry. 500 × 6 → dead.
- Batch nightly : pousse tous les workers, pas de doublon côté MP.
